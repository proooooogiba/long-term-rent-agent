# Evaluation Framework: benchmark + human feedback

Этот документ описывает, как оценивать relocation-агента не только на `demo_stub`, но и на live LLM backend.

Главная идея: использовать двухслойную схему, близкую к практикам production-команд.

- количественный слой: повторяемый offline benchmark на зафиксированном наборе кейсов;
- качественный слой: структурированный human feedback по единым критериям.

Такой подход лучше, чем разовый demo-run, потому что он отделяет:

- воспроизводимость от впечатления “на глаз”;
- качество бизнес-логики от случайного удачного ответа модели;
- регрессии от обычной вариативности LLM.

## Что уже есть в проекте

Базовый quality gate уже реализован:

- `src/evals/metrics.py` считает предметные метрики;
- `src/evals/quality_gate.py` задаёт hard/soft thresholds;
- `python3 -m src.evals.run_qa --enforce-gate` прогоняет offline QA на текущем пайплайне.

Но этого недостаточно для честного сравнения live-моделей, потому что:

- `demo_stub` детерминированный и не показывает реальную вариативность LLM;
- один прогон не показывает стабильность результата;
- без общей rubric user feedback остаётся субъективным.

## Новый benchmark-контур

Для этого добавлен отдельный runner:

```bash
python3 -m src.evals.run_benchmark \
  --llm-backend openrouter \
  --llm-mode required \
  --model deepseek/deepseek-v3.2 \
  --trials 3 \
  --output-json outputs/evals/deepseek-v3.2.json
```

Для быстрого smoke-run можно запускать только часть suite:

```bash
python3 -m src.evals.run_benchmark \
  --llm-backend demo_stub \
  --trials 1 \
  --case-id Q-001 \
  --case-id Q-006
```

Что делает `run_benchmark.py`:

- берёт тот же `data/qa/qa.jsonl` как стартовый benchmark suite;
- поднимает свежую SQLite для каждого trial, чтобы память кейса не “протекала” между независимыми сравнениями;
- внутри trial сохраняет обычную логику кейсов, включая follow-up и replanning;
- запускает каждый кейс несколько раз;
- считает предметные метрики и quality gate для каждого кейса;
- агрегирует результаты в один JSON-report.

## Как сравнивать модели или prompt-варианты

Минимальный честный протокол сравнения:

1. Держать одинаковыми dataset, DB snapshot и runtime-конфиг кроме одной переменной.
2. Менять только один фактор за раз:
   - модель;
   - system prompt;
   - extraction/routing logic;
   - verifier policy.
3. Гонять каждый вариант минимум `3` раза, для финального сравнения лучше `5`.
4. Сравнивать не только средний score, но и устойчивость.

Пример:

```bash
python3 -m src.evals.run_benchmark \
  --llm-backend openrouter \
  --llm-mode required \
  --model deepseek/deepseek-v3.2 \
  --trials 5 \
  --output-json outputs/evals/deepseek-v3.2.json

python3 -m src.evals.run_benchmark \
  --llm-backend openrouter \
  --llm-mode required \
  --model openai/gpt-4.1-mini \
  --trials 5 \
  --output-json outputs/evals/gpt-4.1-mini.json
```

## Какие численные показатели использовать

### 1. Domain metrics

Они уже есть в проекте и остаются базой сравнения:

- `intent_accuracy`
- `required_fields_extracted`
- `budget_constraint_pass_rate`
- `pet_constraint_pass_rate`
- `upfront_cost_correctness`
- `expected_entities_accuracy`
- `recommendation_has_rationale`
- `clarification_correctness`
- `escalation_correctness`
- `info_answer_has_sources`

### 2. Aggregate quality gate

Это release-level индикатор:

- все hard metrics должны пройти;
- доля пройденных soft metrics должна быть не ниже заданного порога.

Он отвечает на вопрос: “можно ли вообще выпускать этот вариант?”

### 3. Mean case pass rate

Средняя доля кейсов, которые проходят case-level gate по всем trial-ам.

Это основной объективный показатель “насколько часто агент справляется”.

### 4. `case_pass_at_k`

Доля кейсов, где хотя бы один из `k` trial-ов был успешным.

Полезно как capability-метрика: показывает, умеет ли модель в принципе решать задачу.

### 5. `case_pass_all_k`

Доля кейсов, которые прошли во всех `k` trial-ах.

Это reliability-метрика. Для production она важнее, чем просто лучший из запусков.

### 6. `case_outcome_consistency`

Доля кейсов, где outcome стабилен между trial-ами:

- либо всегда pass;
- либо всегда fail.

Если значение низкое, значит агент ведёт себя слишком нестабильно, даже если средний score выглядит хорошо.

### 7. `mean_case_score`

Среднее арифметическое по применимым метрикам кейса.

Это удобный общий score для A/B-сравнения, но он не должен заменять hard-ограничения.

### 8. `mean_latency_ms` и `p95_latency_ms`

Нужны, потому что хороший ответ с сильно большей задержкой не всегда лучше продуктово.

## Что считать “победой” варианта

Рекомендуемое правило выбора нового варианта:

- hard-quality gate не должен стать хуже;
- `case_pass_all_k` не должен падать;
- `mean_case_score` должен расти хотя бы на `3-5%`;
- `p95_latency_ms` не должен расти критично;
- human feedback не должен ухудшаться по trust/safety и relevance.

Если вариант улучшает только `pass_at_k`, но не улучшает `pass_all_k`, это хороший research-кандидат, но слабый production-кандидат.

## Capability vs regression

Benchmark лучше разделить на два поднабора.

### Capability suite

Нужен, чтобы искать улучшения.

Сюда входят:

- сложные follow-up кейсы;
- preference conflicts;
- escalation;
- пограничные info-запросы;
- кейсы, где модель раньше ошибалась.

### Regression suite

Нужен, чтобы не ломать то, что уже работало.

Сюда входят:

- базовые `search` кейсы;
- стандартные `clarification`;
- стабильные `info` сценарии;
- критичные safety-кейсы.

Практическое правило:

- capability suite можно улучшать постепенно;
- regression suite должен стремиться к почти полному прохождению.

## Human feedback: как собирать не “впечатления”, а сигнал

Шаблон лежит в `data/qa/human_feedback_template.csv`.

Оценивать нужно не “понравилось / не понравилось”, а по критериям:

1. `task_completion_1_5`
   Насколько ответ реально решил задачу пользователя.
2. `constraint_fit_1_5`
   Насколько агент учёл бюджет, pets, commute, документы и другие ограничения.
3. `relevance_1_5`
   Насколько рекомендации и объяснения относятся именно к запросу пользователя.
4. `clarity_1_5`
   Насколько ответ понятный, структурированный и пригодный к действию.
5. `safety_trust_1_5`
   Не сделал ли агент опасных допущений и можно ли ему доверять в этом ответе.
6. `would_use_again_1_5`
   Готов ли пользователь снова использовать такого агента.
7. `critical_issue_yes_no`
   Был ли критический сбой, из-за которого ответ нельзя считать приемлемым.

### Как агрегировать human feedback

Для защиты и для продуктовой оценки достаточно 4 агрегатов:

- `mean_human_score`: среднее по всем шкалам 1-5;
- `top2box_rate`: доля оценок `4-5`;
- `critical_issue_rate`: доля ответов с `critical_issue_yes`;
- `nps_like_reuse_rate`: доля `would_use_again >= 4`.

### Минимальные правила интерпретации

- если `critical_issue_rate > 0`, вариант нельзя считать безопасным по умолчанию;
- если `constraint_fit` заметно ниже `relevance`, агент “говорит красиво, но не по делу”;
- если `task_completion` ниже `clarity`, значит ответ хорошо написан, но плохо решает задачу.

## Как встроить это в процесс проекта

### Перед изменением модели / prompt

- прогнать текущий benchmark и сохранить JSON-report;
- зафиксировать baseline.

### После изменения

- прогнать тот же benchmark повторно;
- сравнить aggregate gate, `pass_all_k`, `mean_case_score`, latency;
- прочитать транскрипты упавших кейсов.

### После demo или pilot usage

- собрать user feedback по шаблону;
- добавить реальные неудачные кейсы обратно в `data/qa/qa.jsonl`;
- перевести повторяющиеся проблемы в новые regression cases.

## Что говорить на защите

### Короткая версия

- demo нужен, чтобы показать сценарий работы агента, но demo сам по себе не является доказательством качества;
- качество мы оцениваем отдельно через повторяемый benchmark на live LLM;
- дополнительно мы собираем user feedback по фиксированной rubric, чтобы проверить, что численные метрики совпадают с восприятием пользователя.

### Основной тезис для ответа

“Мы специально разделили демонстрацию и оценку качества.  
Demo показывает, как агент работает, а качество мы проверяем отдельно: количественно через повторяемый benchmark на одном и том же наборе кейсов и качественно через структурированный user feedback. Это ближе к индустриальному подходу, чем разовый успешный demo-run.”

### Развёрнутый ответ на 30-40 секунд

“Главная проблема агентных систем в том, что один красивый запуск ничего не доказывает, особенно если используется live LLM с недетерминированным поведением. Поэтому мы не оцениваем агента по demo. Demo нужен только для визуализации пайплайна и пользовательского сценария.  

Для оценки качества у нас два слоя. Первый слой — количественный benchmark: мы берём фиксированный набор кейсов, запускаем один и тот же агент несколько раз на live backend и считаем предметные метрики, quality gate и показатели стабильности, например `pass@k`, `pass_all_k` и consistency. Это позволяет сравнивать модели, prompt-варианты и изменения в логике агента не по впечатлению, а по воспроизводимым числам.  

Второй слой — качественная проверка через user feedback rubric. Пользователь или reviewer оценивает не абстрактно “понравилось / не понравилось”, а по критериям: решил ли агент задачу, учёл ли ограничения, был ли ответ релевантным, понятным и безопасным. В итоге мы получаем и численную, и пользовательскую оценку, и можем смотреть, совпадают ли они между собой.”

### Если спросят, почему это объективнее обычного demo

- используется фиксированный dataset, а не случайные запросы “с головы”;
- для сравнения меняется только один фактор за раз: модель, prompt или часть логики;
- каждый кейс запускается несколько раз, чтобы учесть вариативность LLM;
- benchmark поднимает свежую SQLite на каждый trial, чтобы состояние и память не искажали сравнение;
- решение о качестве принимается не по одной метрике, а по комбинации hard-gate, soft metrics и reliability;
- user feedback тоже формализован, а не собирается в виде свободных комментариев без критериев.

### Если спросят, какие именно показатели мы считаем главными

- для корректности бизнес-логики: `budget_constraint_pass_rate`, `upfront_cost_correctness`, `clarification_correctness`, `escalation_correctness`;
- для общего качества ответа: `intent_accuracy`, `required_fields_extracted`, `expected_entities_accuracy`, `recommendation_has_rationale`, `info_answer_has_sources`;
- для стабильности live LLM: `mean_case_pass_rate`, `case_pass_at_k`, `case_pass_all_k`, `case_outcome_consistency`;
- для пользовательского качества: `task_completion`, `constraint_fit`, `relevance`, `clarity`, `safety_trust`, `would_use_again`.

### Если спросят, когда мы считаем новую версию агента лучше

Мы считаем новую версию лучше, если:

- она не ломает hard-quality gate;
- не ухудшает `case_pass_all_k`, то есть надёжность;
- улучшает `mean_case_score` или `mean_case_pass_rate`;
- не создаёт роста `critical_issue_rate` по human feedback;
- не становится заметно хуже по latency без продуктовой причины.

### Чего не стоит утверждать на защите

- что один удачный demo подтверждает качество агента;
- что средний score сам по себе достаточен без проверки стабильности;
- что user feedback можно заменить только автоматическими метриками;
- что benchmark полностью заменяет чтение trace и разбор failure cases.

### Самая короткая формулировка

“Мы оцениваем агента в двух плоскостях: численно через повторяемый benchmark на live backend и качественно через формализованный user feedback rubric. Demo у нас отвечает за наглядность, а не за доказательство качества.”
