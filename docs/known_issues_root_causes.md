# Проблемы и причины

Краткая сводка по основным сбоям, которые проявлялись в демо и интеграции.

| Проблема | Как проявлялось | Причина | Статус |
| --- | --- | --- | --- |
| `LLM routing failed: Connection error` | UI падал с traceback вместо ответа | Live-вызов в OpenRouter мог временно отваливаться по сети, а ранний код не имел мягкого fallback-сценария | Исправлено: добавлен fallback на demo-LLM для routing/intake/replanning/final composition |
| Не распознавался `housing_type` | Агент повторно спрашивал `комнаты или тип жилья`, даже когда пользователь уже написал `квартира` | Поле `housing_type` было слабо нормализовано, rule-based extraction почти отсутствовал, а сам признак не доходил до retrieval | Исправлено: добавлена нормализация и прокидывание в search/SQL |
| Бюджет иногда брался из даты | Для строки вида `07.07.2026, 1000 usd` бюджетом становился `2026` | Fallback-parser для бюджета забирал первое подходящее число и не защищался от дат | Исправлено: бюджет сначала ищется как сумма с валютой, дата исключена |
| Москва не находилась даже при включённом `cian` | Поиск возвращал `0 candidates`, агент просил ослабить критерии | Локальная БД не содержит Москву; если внешний MCP не работает, поиск остаётся только на demo-каталоге | Частично исправлено: путь через `cian` включён, но зависит от внешнего провайдера |
| Внешний `cian` вообще не поднимался | В warning было `Install mcp>=1.27,<2` | В активном Python-окружении отсутствовал пакет `mcp` | Исправлено: зависимость установлена в тот интерпретатор, которым запущен Streamlit |
| `2000 usd` по Москве давали пустой результат | `cian` отвечал пусто даже при живом провайдере | Бюджет пользователя в USD уходил в `cian` как `2000 RUB`, потому что не было конвертации между валютой кейса и валютой провайдера | Исправлено: добавлен `budget_currency` и конвертация budget/request/result |
| `cian` ломался именно внутри собранного приложения | Изолированный search работал, а в полном agent graph появлялся `ConnectError`/`TaskGroup` | `GraphDependencies` подхватывал не проектный `.env`, а домашний `/Users/iopogiba/.env`, где был прописан битый proxy `prx-squid-rev:9090` | Исправлено: `.env` теперь читается только из локального проекта |
| UI мог падать на финальном ответе даже при живом backend | `final_composer` падал на validation error JSON-секции clarification/recommendation | В UI-сессии агент фактически жил в `required`-режиме из конфига, поэтому не уходил в demo-fallback при кривом JSON от live LLM | Смягчено: Streamlit-сессия переведена в `auto` для более безопасного fallback |

## Итог

Главные причины проблем были не в одной точке, а на стыке трёх слоёв:

1. `LLM extraction/formatting`
2. `MCP integration / environment`
3. `currency + retrieval contract`

Самые критичные интеграционные причины:

- внешний `cian` требовал установленный `mcp`
- приложение подхватывало чужой `.env` с невалидным proxy
- бюджет в USD не конвертировался в валюту провайдера

Самые критичные продуктовые причины:

- `housing_type` не был полноценно протянут end-to-end
- fallback-budget parser ошибочно цеплялся за год из даты
